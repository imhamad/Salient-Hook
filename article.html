I Spent a Week Inside Hacked WordPress Sites. Here's What I Found and What I Built to Stop It.

By Hamad K  ·  hamadhere.de

We had several WordPress sites get hit within the same week. Clients were reporting a strange Cloudflare-looking popup appearing on their sites asking visitors to "verify" before continuing. At first glance it looked almost convincing — the kind of thing a non-technical visitor wouldn't question.

After the third site reported the same thing I knew something bigger was going on.

I started going through each codebase manually. Not with a scanner, not with an automated tool — line by line, file by file, through theme folders, plugin directories, wp-content, everything. Across multiple sites. It took days.

What I kept finding was a plugin sitting in the plugins folder called WP Performance Analytics. Author listed as "Developer Tools Team", version 1.2.95, description reading "Lightweight site performance monitoring, page load analytics and optimization suggestions." Nothing about it screamed malicious. It looked like something a client or a previous developer had installed and forgotten about.

On one of the sites I disabled it as a test. The popup disappeared immediately.

That was it. That was the culprit.

The popup itself wasn't just annoying — it was dangerous. It was engineered to trick Windows users into pressing Win + R and pasting a command into the Run dialog. That command reached out to an external server and silently downloaded malware in the background. Credentials, browser sessions, crypto wallets — all of it potentially exposed. Sucuri documented the full technical breakdown of this campaign here: https://blog.sucuri.net/2025/05/another-fake-cloudflare-verification-targets-wordpress-sites.html

The frustrating part wasn't finding it. It was realising how easily it got in and how long it had potentially been sitting there.

WordPress by default allows any administrator to install plugins. That's fine in principle. In practice, if credentials get compromised — even briefly — an attacker can drop a plugin, close the session, and walk away. The plugin does its job quietly, the credentials get changed or nothing seems obviously wrong, and nobody notices until visitors start reporting something strange.

That's the gap I wanted to close.

I built Salient Hook to do a few specific things. First, it locks down plugin installation entirely — the Add New Plugin screen is gone, the upload routes are blocked, ZIP files can't come in through the media uploader. Second, it freezes plugin updates so nothing is pulling in changes from external sources without a deliberate decision to allow it. Third, and most directly relevant to this incident, it actively scans for WP Performance Analytics and anything else matching the same code fingerprint. If it finds it active, it kills it immediately and puts a notice on screen.

There's also a theme scanner built in because this particular malware doesn't just live in the plugin — it drops a file called verification.html into theme folders and injects code into header.php. Even if you delete the plugin, the artefacts can remain. The scanner checks every installed theme for those files, flags the exact line where something suspicious was found, and tells you whether it's a confirmed threat or just worth investigating.

Everything is visible from a single dashboard under Settings in WordPress. Green means locked, red means something needs attention. No clutter, no upselling, no noise.

The plugin is open source and available on GitHub: https://github.com/imhamad/Salient-Hook

If you're managing multiple WordPress sites — especially for clients — it's worth having something like this in place before you need it. The week I spent going through those codebases manually is not a week I want to repeat.
